# Reaction Standee Decision Log

開発中の判断理由を後から見返すためのメモです。README は現在の使い方、このファイルは経緯と意図を残します。

## 2026-06-22: 公開版はGitHub Pagesを主軸にする

### 背景

Reaction Standee は、自分のYouTube制作で使う録画ツールとして成立してきた。次の段階では、転職ポートフォリオとして見せられるように、GitHub上で触れる公開版を用意したい。

### 判断

公開版は GitHub Pages で配信する静的Webアプリにする。アプリ名とリポジトリ名は、引き続き `Reaction Standee` / `reaction-standee` を使う。

GitHub Pagesでは直URLの `/record` が404になりやすいため、公開版の画面切り替えは `?route=record` のようなクエリルートを使えるようにする。

### 理由

- `Reaction Standee` は、リアクションする立ち絵パネルという実態に合っている。
- GitHub Pagesなら、採用担当者や知人がURLからすぐに動作を確認できる。
- npmコマンドやローカルサーバ起動を一般ユーザーに要求せずに済む。
- WKWebView録画ウィンドウは自分のMacでの高品質録画向けとして価値があるが、一般公開の主導線にすると配布、署名、公証の負担が大きい。

### 注意点

- GitHub Pages公開版では、ローカル開発用の `/api/settings` や `/api/assets` は存在しないため、ブラウザ内保存を基本にする。
- PWA対応は、GitHub Pages公開版をDockに追加しやすくするための次段階として扱う。
- 本気の録画用途では、引き続き自分用のWKWebView録画ウィンドウが一番扱いやすい。

## 2026-06-21: 録画方式はDOM表示を主軸にする

### 背景

Reaction Standee は、ショート動画向けに立ち絵を自然に表示し、ポーズ、口パク、まばたき、生命感演出を組み合わせるアプリとして進めている。

録画方法として、以下を検討した。

- OBS Browser Source
- Safari/Chrome のウィンドウキャプチャ
- Canvas に再描画してブラウザ内録画
- Electron によるMacアプリ化
- 将来的なWebGL/PIXI.jsなどの専用描画エンジン

### 判断

短期の主軸は、DOM/CSS版の高品質表示をそのまま使うことにした。

具体的には、設定は `/settings`、録画表示は `/record`、Macアプリ レベル1では Electron ウィンドウ内に `/record` を表示する。

### 理由

- DOM/CSS版の見た目品質にすでに満足できている。
- 白フチ、差分オーバーレイ、生命感エフェクト、背景表示などはDOM/CSSの方が作りやすく、見た目も安定している。
- Canvas版は録画イメージとしては分かりやすいが、白フチやリアクションエフェクトの見た目がチープになりやすい。
- CanvasでDOMと同じ見た目を再現しようとすると、実質的に別描画エンジンを作ることになり、目的から外れやすい。
- OBS Browser Source は環境によってCSSアニメーションや画像切替が重くなる。
- Electron レベル1なら、ブラウザのツールバーを避けつつ、DOM/CSSの見た目を維持できる。

### 残すもの

- `/record`: 現在の主導線。DOM/CSS品質の録画表示。
- `/canvas`: Canvas出力の実験画面として残す。短期の主導線にはしない。
- `/capture`: 旧来のブラウザウィンドウキャプチャ用として残す。主導線からは外す。
- `/avatar`: OBS Browser Source用の代替表示として残す。主導線からは外す。

## 2026-06-21: 画像と設定は共有ローカル保存へ寄せる

### 背景

Safari、Chrome、OBS Browser Source、Electron は保存領域が分かれるため、IndexedDBやlocalStorageだけだと画像や設定を再登録する必要が出る。

### 判断

画像は `.reaction-standee/assets/`、表示設定は `.reaction-standee/settings.json` に共有保存する。

### 理由

- 初回だけ `/settings` を開けば、既存ブラウザ保存から共有保存へ移せる。
- Electron版が同じ画像と設定を読める。
- Git管理には入れず、ローカル実データとして扱える。
- 将来、より本格的なローカル保存APIへ移行する場合も入口を一本化しやすい。

### 注意点

- `.reaction-standee/` はGit管理外。
- 別Macへ移す場合は `.reaction-standee/` も移す必要がある。
- 画像差し替えが即時に見えない場合は表示側を再読み込みする。

## 2026-06-21: 録画用ウィンドウはSafari Webアプリを主軸に戻す

### 背景

`/record` を録画しやすくするため、以下の方法を実際に比較した。

- Electronで `/record` を表示する
- Electronで `/record` を表示専用にして、`/settings` から状態同期する
- Electronで `127.0.0.1:5173/record` を直接開く
- Chrome/Braveのアプリモードで `/record` を開く
- Safariの「Dockに追加」で `/record` をWebアプリ化する
- Canvas録画を使う

### 観察

- Safari上の `/settings` と `/record` は、キャラの動き、口パク、生命感エフェクト、リアクションエフェクトが比較的なめらか。
- Chrome/Braveアプリモードは、キャラの動きが重く、Electronでエフェクトが弱い時と近い体感だった。
- Electronは、最初はカメラ/MediaPipe/マイク解析が二重に走っていたため重かった。
- Electronを表示専用にして二重処理を止めても、Safariに比べるとコマ送り感や遅延が残った。
- Electronで `/settings` から `/record` へ状態同期する方式は、ポーズ切り替えが一段遅れ、エフェクト発火のタイミングもズレやすい。
- Electronが `127.0.0.1:5173/record` を直接開くようにしても、Chromium系の描画負荷は残った。
- Canvas録画は録画範囲を制御しやすいが、白フチやリアクションエフェクトの見た目がチープになりやすい。
- Safari Webアプリ化は、描画品質と軽さが最も良かった。ただし、録画時のクロップ問題は残る。

### 判断

短期の実用録画は、SafariのWebアプリ化またはSafari上の `/record` を主軸にする。

Electron録画ウィンドウ案は保留する。Chrome/Braveアプリモードも、描画が重いため主導線にしない。Canvas録画は実験用途として残す。

### 理由

- 最終的に重要なのは「録画された見た目が自然でなめらかであること」。
- Safariが最も軽く、DOM/CSSの見た目品質も維持できる。
- ElectronやChromeは録画用ウィンドウとして扱いやすいが、描画の重さが本質的な問題になった。
- Electron単体アプリに入力処理を持たせれば同期遅延は消せるが、Electron/Chromiumの描画負荷が消える保証はない。
- いまElectronを磨き続けると、録画したいという本題からズレやすい。

### 今後の方針

- `/record` のDOM/CSS表示品質を主軸として維持する。
- 録画運用はSafari Webアプリを第一候補にする。
- クロップ問題は、OBS側のシーン/ソース設定を保存する、またはmacOS録画側の運用で最小化する。
- Electronは、現時点では録画主導線から外す。
- Macアプリ化を再検討する場合は、「Safari入力 + Electron表示同期」ではなく、入力・設定・表示が同じアプリ内で完結する構成として別途検証する。
- ただし、その検証ではまずElectron/Chromium描画がSafari並みに軽くなるかを最小構成で確認する。

### 残る課題

- Safari Webアプリで録画する場合、ウィンドウ上部や余白をどう扱うか。
- OBSまたはmacOS標準録画で、毎回ずれないクロップ運用をどう作るか。
- 将来的に本格Macアプリ化するなら、Safari並みの描画品質と軽さをどう維持するか。

## 2026-06-22: 録画環境改善はWKWebView検証を最優先にする

### 背景

SafariのDOM/CSS表示は軽く、見た目品質も良い。一方で、通常のSafariウィンドウやSafari Webアプリでは、録画時にウィンドウ上部や余白の扱いが残り、クロップ調整が面倒になる。

Electronはアドレスバーやタブのない9:16録画ウィンドウを作りやすかったが、Chromium系の描画負荷が残り、Safariほど滑らかにはならなかった。

Chromeで簡易計測したところ、`/record` は重いが、表示専用に近い `/avatar` は60fps付近まで出た。つまり、DOM/CSS表示そのものが常に重いというより、カメラ/MediaPipe/音声入力/白フチ/背景合成などが組み合わさった時にChrome系で詰まりやすい可能性が高い。

### 現在のコード実態

- `/settings` は設定、画像登録、プレビュー、カメラ/音声入力、デバッグを担当する。
- `/record` は録画用表示だが、現在は `/avatar` 以外で入力処理を動かす実装のため、カメラトラッキングと必要に応じた音声入力も持つ。
- `/avatar` は表示専用に近く、ローカルAPIからリアクション状態を受け取る。
- `/canvas` はCanvas録画の実験場として残す。
- Electron実験差分は `electron-experiment` ブランチへ退避済み。本線の録画導線にはしない。

### 判断

次の最優先は、Swift + WKWebViewで `/record` を表示する最小Mac検証にする。

目的はMacネイティブ化ではなく、Safariに近いWebKit環境で、アドレスバーやタブのない9:16録画用ウィンドウを作れるかを確認すること。

### 理由

- WKWebViewはWebKit系なので、Electron/ChromeよりSafariに近い描画特性を期待できる。
- 既存WebアプリをSwiftへ移植せず、`http://127.0.0.1:5173/record` を読み込むだけなら検証コストが比較的小さい。
- うまくいけば、Safariの滑らかさに近いまま、Electronのような録画しやすいウィンドウを得られる。
- 失敗しても、Webアプリ本体を壊さずに撤退しやすい。

### 検証すること

- WKWebViewでローカルの `/record` を表示できるか。
- WKWebView内で `navigator.mediaDevices.getUserMedia` が使えるか。
- MediaPipeのWASM/モデル読み込みとGPU delegateが動くか。
- カメラ許可に `NSCameraUsageDescription` が必要か。
- 口パクや音声入力も使う場合、`NSMicrophoneUsageDescription` が必要か。
- `WKUIDelegate` のメディア許可処理が必要か。
- 9:16のウィンドウサイズで起動できるか。
- 余計なUIなしで、macOSのウィンドウ録画対象として扱いやすいか。
- Safari Webアプリ、Electron、Chromeと比べて体感が軽いか。

### 今回は対象外

- 既存Webアプリ本体のSwift移植。
- Macアプリ側での録画機能実装。
- App Store配布、署名、公証、notarization対応。
- 本格的なMacネイティブUI化。

### 次点の軽量化方針

WKWebView検証と並行して、Chrome/Electron向け軽量化は別タスクとして残す。

候補は以下。

- `/record` を表示専用化し、入力を `/settings` に寄せる。
- カメラ解像度、カメラfps、MediaPipe推論fpsを下げる。
- 毎フレーム推論せず、推論中に次の推論を重ねない。
- Chrome/Electron向けに軽量モードを用意する。
- 白フチの多段 `drop-shadow` を軽量化する。
- 白フチ付きPNGの事前生成を検討する。
- 画像APIを全アセット一括取得ではなく、必要な画像だけ取得する形にする。

Canvas/WebGL/PixiJSは、現時点では本線に戻さない。描画ボトルネックが明確になり、見た目を大きく損なわずに改善できる見込みがある場合だけ、別ルートで小さく比較する。

### 将来の公開方針

他の人にも使ってもらう場合、最初はWebアプリ公開を優先する。

- GitHubにコードを置き、READMEを整える。
- GitHub Pagesなどで静的Webアプリとして公開できるか検討する。
- カメラ映像はブラウザ内で処理し、サーバーへ送信・保存しない設計を維持できるか確認する。
- PWA化は必要性を見て検討する。
- Electron/Macアプリ配布は、署名や公証の負担があるため優先度を下げる。

公開時には、カメラ映像、画像保存、ローカル処理、外部送信の有無について、実装に合わせて正確に説明する。

## 2026-06-22: WKWebView検証ではカメラ入力を軽くする

### 観察

WKWebViewで `/record?perf=1` を実測したところ、初期条件ではおおむね以下だった。

- 1280x720カメラ、毎フレーム推論: 約43fps、推論約20ms
- 白フチOFF: 約43fpsで大きく変わらず
- 背景、エフェクト、生命感、差分オーバーレイ、白フチをまとめてOFF: 約46fps
- 640x360カメラ、推論30fps制限: 約55fps

### 判断

WKWebView録画ウィンドウの標準起動は、`/record?camera=low&inferFps=30` にする。

### 理由

- 白フチやCSSエフェクトを切っても改善幅は小さく、主な詰まりはMediaPipe推論とカメラ入力側に見えた。
- 録画されるキャラクター画像は登録PNGとDOM/CSS表示であり、カメラ解像度を下げても録画画質そのものは下がらない。
- ポーズ認識は反応コマンド用途なので、30fps推論でも実用上は十分な可能性が高い。
- WKWebView自体は、入力負荷を落とすと55fps付近まで上がり、Electron/Chromeより有望に見える。

### 次に見ること

- 640x360/30fpsでポーズ切り替えの反応が遅く感じないか。
- 口パク、生命感、リアクションエフェクトが実録画で自然に見えるか。
- Safari Webアプリ運用と比べて、クロップしやすさ込みでWKWebViewを採用する価値があるか。

## 2026-06-22: WKWebViewをMac録画の推奨ルートにする

### 観察

WKWebView録画ウィンドウで実録画を確認した。

- キャラクター動作、ポーズ切り替え、生命感エフェクトは実用上問題なかった。
- ウィンドウ上部に透明のドラッグ領域を置き、`performDrag` で明示的に移動できるようにした。
- Safari通常ウィンドウより、ブラウザツールバーや余白のクロップ問題が大きく減った。
- macOSのウィンドウ録画で、映像のクロップは実用的な範囲になった。
- 口パク用マイクをONにすると、当初はmacOS画面録画側にユーザーの声が入らなかった。
- 口パク用マイク取得時の `echoCancellation`、`noiseSuppression`、`autoGainControl` をOFFにしたところ、口パクとmacOS録画のマイク収録を両立できた。
- `ReactionStandeeWKPreview.app` はFinderやSpotlightから起動できるが、Webアプリ本体と開発サーバは内蔵していない。
- macOS画面録画では、アプリ側でカーソルを隠しても録画結果にカーソルが残ることがあった。

### 判断

短期のMac録画主導線は、`/settings` で設定し、WKWebView録画ウィンドウで `/record` を表示して、macOS画面録画またはOBSのウィンドウキャプチャで収録する方式にする。

現時点の `.app` は、単体で完結するMacアプリではなく、ローカルで起動中のWebアプリを録画しやすい9:16ウィンドウとして開くための軽量ラッパーとして扱う。

### 理由

- DOM/CSS版の見た目品質を維持できる。
- Canvas版のように白フチやエフェクトを再実装しなくてよい。
- Electron/Chromeより動作が軽く、Safariに近いWebKit系の挙動を使える。
- Safari通常ウィンドウより録画範囲を合わせやすい。
- カメラ入力を640x360、推論を30fpsに抑えても、録画されるキャラ画像の品質は下がらない。
- 口パク用マイクと録画用マイクも共存できる見通しが立った。
- サーバ起動まで `.app` に抱え込むとハマりどころが増えるため、まずは録画ウィンドウとして安定させる。

### 残す課題

- `npm run dev -- --host 127.0.0.1` は別途起動が必要。
- サーバ未起動時は画面が分かりづらくなるため、GUI起動を整えるなら未起動案内を改善する。
- `ReactionStandeeWKPreview.app` は開発用の簡易 `.app` であり、配布用の署名、公証、アイコン整備は未対応。
- 将来的に完全なGUI起動にするなら、`.app` からローカルWebサーバも起動する方式を検討する。
- マウスカーソルは録画前にWKWebViewウィンドウ外へ移動する運用にする。アプリ側で無理に隠すと、閉じる操作などがやりづらくなる。

## 2026-06-29: ローカル版と公開版を一つのアプリとして維持する

### 状況

GitHub Pages公開後も、ローカル制作環境では共有画像保存、WKWebView録画、OBS向け状態同期が必要になる。一方、GitHub Pagesは静的ホスティングのため、Viteの `/api/reaction`、`/api/assets`、`/api/settings` は存在しない。

ローカル版と公開版を別々のアプリとして分岐させると、カメラトラッキング、表示、ポーズ判定、口パク、生命感エフェクトの修正を二重に反映する必要が生じる。

### 判断

機能本体は一つのReactアプリとして維持する。環境差は、ビルド時の `VITE_DEPLOY_TARGET` で判定する小さな境界に限定する。

- 通常のローカル起動では、Viteの共有ローカルAPIを有効にする。
- `npm run build:pages` では `VITE_DEPLOY_TARGET=static` を設定し、共有ローカルAPIへの通信を行わない。
- GitHub Pages版の画像と設定はIndexedDBとlocalStorageに保存する。
- PWAのService Workerは静的公開ビルドだけで登録する。
- カメラ、MediaPipe、表示、ポーズ判定、口パク、生命感エフェクト、設定UIは共通コードを使う。

### 理由

- 日常的に変更する機能を一か所で保守できる。
- ローカル録画環境を失わず、公開デモも同じ品質で改善できる。
- 存在しないAPIへの404通信やSSE再接続を公開版から除去できる。
- 将来、GitHub Pages以外へ静的配信する場合も同じ公開ビルドを再利用しやすい。

## 2026-06-29: GitHub Pages版をPWA対応する

### 判断

公開版にWeb App Manifest、Service Worker、アプリアイコンを追加する。PWA登録は静的公開ビルドだけで有効にし、ローカル制作・WKWebView録画の動作には介入しない。

### 目的

- Safariの「Dockに追加」やモバイルの「ホーム画面に追加」で起動しやすくする。
- ブラウザの通常タブよりアプリに近い表示を提供する。
- 一度取得したアプリ本体の静的ファイルをキャッシュし、再表示を安定させる。

### 制約

- MediaPipeモデルなど外部配信リソースがあるため、完全オフラインは保証しない。
- PWAはWKWebView録画ウィンドウと同じ厳密な9:16固定・クロップ解消を保証するものではない。
- オリジナルのデモキャラクター素材とREADMEの動作GIFは別作業として追加する。

## 2026-06-29: SafariとDock版の設定移行はファイルで行う

### 観察

macOSのSafariでGitHub Pages版を設定してから「Dockに追加」しても、登録画像と設定はDock版へ引き継がれなかった。SafariのWebアプリはCookie以外のlocalStorageやIndexedDBをコピーせず、追加後もSafari本体とWebアプリの保存領域を共有しない。

### 判断

- Dock版は初回設定が必要なため、設定画面から起動する。
- 設定、キャラクター画像、背景、目線、まばたき、口パク画像を一つのJSONファイルへ書き出し、別環境で読み込めるようにする。
- カメラとマイクのdeviceIdは環境間で再利用できないため、読み込み時は既定デバイスへ戻す。
- 公開版の設定画面から録画表示へ同じウィンドウで移動し、録画表示では `Esc` キーで設定へ戻れるようにする。
- PWA内の設定画面と録画表示はページ再読み込みを挟まず切り替え、起動済みのカメラと音声解析を維持する。
- ローカル版の共有APIは、Safari、WKWebView、OBS間の自動共有用途として引き続き維持する。

### 理由

- 静的なGitHub Pagesだけで、ブラウザが分離した保存領域を直接共有することはできない。
- 大容量の画像をURLやCookieへ含める方法は、容量と安全性の面で適さない。
- ファイル移行ならサーバーへ画像を送らず、利用者自身の端末内で完結できる。
